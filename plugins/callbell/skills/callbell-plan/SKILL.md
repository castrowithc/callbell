# /callbell-plan

An idea goes in, a project and its work packages come out. After that this session is done, and the work grows from there in conversation.

The user started this deliberately, so the intent to build is settled: don't reopen whether they want it. What's open is *what* they want, and that's the work below.

Planning is a conversation, not a form. Never hand the user a template to fill in. The checklist below is yours: check what they gave you against it, and ask about what's missing.

## Ask, all through this phase

Questions are the point of this phase, not its failure. Don't guess your way to a plan that reads well and means nothing.

- **Name your assumptions** instead of burying them. If a requirement has two readings, put both up for choice and let the user decide; don't silently pick one.
- **Push back** on anything unclear or contradictory. A plan built on a misunderstanding costs more than the question would have.
- **Only cosmetic decisions are yours** during planning (wording, naming, ordering that changes nothing).

## 1. Clarify what kind of idea this is

Do this first; it decides how you work:

- **Mature in the user's head:** draw it out, don't co-invent. Ask, reflect back, and keep your own ideas out unless asked. Adding here drives the result away from what they actually want.
- **Half-baked:** genuinely think along. Offer options, name trade-offs, bring a recommendation.
- **Generated by another agent** (a plan, spec, or analysis handed to you): be most skeptical here. That's checked plausibility, not checked intent, and it reads more finished than it is. Walk it through with the user instead of adopting it.

## 2. Clarify these axes before you cut

Your checklist, not the user's. Ask about what their input doesn't already cover:

- **Outcome:** what exists afterward that doesn't now?
- **Why:** what's it for? Without this no package can decide anything on its own.
- **Out of scope:** what explicitly does not belong? The boundary is the useful half.
- **What already exists:** what gets reused instead of built? Look in the repo; don't assume a greenfield.
- **Verification:** how is each outcome checked? Something observable, not "looks right".
- **Risk:** what's unclear, what could make the plan moot?

## 3. Cut into work packages

A package is a task file (`task-<slug>.md`, template in `__callbell__/templates/`), carrying why, scope, approach, and a definition of done: enough to work on with no further questions. It's not a list of intermediate states ("step 3 of 7"); each package says "here's everything you need".

- **Size it to one session**, before context runs out. That's the span in which an agent plans and works coherently. If you can't estimate the size, the work isn't understood: break it down.
- **A package never names another package.** Order and dependency go solely in the roster (`BACKLOG.md` or the project's `index.md`). A later package doesn't read an earlier one: it needs its *result*, and that's in the repo. Writing a filename into a task body *is* the instruction to read it, so don't.
- **Several related packages** are a reason to propose a project folder `__callbell__/backlog/<project>/`. Propose it, explain it in a sentence, and create it only after approval. Group inside a large project with headings in the `index.md` (backend, frontend). A heading is grouping enough; it never becomes a file level.
- New packages start `status: draft`. Assumptions die in draft: shape them with the user until they hold, then move to `active`.

## Done

When the packages stand and no questions are open, this session is over: say so and stop planning. From here the work grows in conversation, and re-cutting an active package is normal (say so when you do, never silently).

Stopping is behavior, not a switch: this guidance stays in context until it's compacted away, so stop planning even though the text remains.
